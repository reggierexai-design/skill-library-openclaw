# AI Guardrail Review ## Purpose - Review an agent for risky behaviors, abuse paths, and missing refusals before wider use. - This is the pre-launch safety gate. An agent that works well in testing can behave catastrophically in production when users push it outside its intended use. This skill systematically finds those paths before users do. ## Use when - Use before launching or widening access to an agent that can browse, execute, or message users. - Use after any prompt, tool, or architecture change that could affect safety boundaries. - Use when the agent is being given new capabilities (tool access, user messaging, file system access, code execution). ## Avoid when - Do not confuse a style review with a real guardrail review — this is about safety, not tone. - If the agent has already caused a safety incident, contain first (`safe_live_systems`), then review. - Do not use this as a substitute for a formal security review (`sec_threat_model`) when the agent handles sensitive data or has production access. ## Inputs to gather - Agent's full capability surface: what tools it can call, what APIs it can reach, what data it can access, what actions it can take on behalf of users. - Current prompt with all instructions, constraints, and refusal rules. - Target user population: trusted internal users, authenticated external users, or the open internet? - Known abuse vectors for similar agents: prompt injection, jailbreaking, data exfiltration, unauthorized actions. - Compliance requirements: PII handling, content policy, industry regulations. - Existing incident history: any past safety failures with this agent or similar systems. ## Operating rules - **Model helpfulness is never an excuse for unsafe defaults.** The model will try to comply with every request. Guardrails exist because compliance is not always appropriate. - **Check misuse, not just intended use.** For every capability, ask: "What's the worst thing a user could do with this?" Then verify the guardrail actually blocks it. - **Prefer design constraints over brittle warnings.** "Don't do X" in the prompt is a warning. Removing the tool that enables X is a constraint. Constraints survive adversarial input; warnings don't. - **Test the guardrails, don't just describe them.** For every guardrail, write an adversarial test case that tries to bypass it. A guardrail that hasn't been tested is a guardrail that doesn't work. - **Map the escalation path.** When a guardrail triggers, what happens? Silent refusal? User notification? Operator alert? Each response has different security and UX implications. ## OpenClaw tool pattern - Read the full prompt, tool schemas, and runtime config before reviewing. You can't review guardrails you can't see. - Test guardrails against adversarial inputs, not just the intended user queries. Use the eval harness to automate this. - When recommending changes, distinguish between prompt-level guardrails (fast to implement, fragile) and architecture-level guardrails (slower to implement, robust). - After implementing guardrails, add test cases to `ai_eval_harness` to prevent regression. ## Expanded workflow 1. **Map the agent's capability surface.** List every action the agent can take: API calls, file operations, user messages, data access, code execution. For each, note the blast radius if misused. 2. **Enumerate credible misuse paths.** For each capability, brainstorm: prompt injection (can a user trick the agent into unintended actions?), data exfiltration (can the agent be made to reveal sensitive data?), unauthorized escalation (can the agent be pushed beyond its intended permissions?), reputation damage (can the agent generate harmful or embarrassing content?), resource abuse (can the agent be made to spend money or time wastefully?). 3. **Evaluate existing guardrails.** For each misuse path, check: is there a guardrail? Does it actually block the misuse in practice (test it)? Is it a prompt-level warning or an architecture-level constraint? 4. **Recommend controls.** For each unguarded or weakly guarded path, propose: a specific guardrail, the implementation layer (prompt vs. tool vs. architecture), and a test case to verify it works. 5. **Rank by risk.** Score each misuse path: likelihood × impact. Fix the highest-risk paths first. Mark launch blockers (any path with high impact and no guardrail). 6. **Produce the review document.** Compile: capability map, misuse paths, guardrail status, recommendations, risk ranking, launch blockers, and test cases. 7. **End with a clear launch decision.** Is the agent safe to launch? Safe with conditions? Not safe until specific fixes are made? ## Output contract - Capability surface map with blast radius per capability - Misuse path enumeration with likelihood × impact scoring - Guardrail status: which paths are guarded, which aren't, which guards are fragile - Recommended controls with implementation layer (prompt / tool / architecture) - Risk ranking: launch blockers, high priority, medium priority, low priority - Test cases for each recommended guardrail - Clear launch recommendation: go / go with conditions / no-go ## Failure modes to avoid - **Relying on prompt-only guardrails for high-impact paths.** "Don't do X" in the prompt will fail under adversarial input. High-impact paths need architecture-level constraints. - **Reviewing intended use instead of misuse.** The agent was designed for intended use — it will probably work fine there. The review must focus on what happens when users push outside the intended path. - **Skipping the test.** Recommending a guardrail without testing it means you don't know if it works. Always write the adversarial test case. - **Over-guarding low-risk paths.** Every guardrail has a UX cost. If the user experience degrades significantly for a low-risk path, the guardrail may be worse than the risk. - **Ignoring indirect abuse paths.** Sometimes the dangerous capability isn't the obvious one. An agent that can send emails can be used for phishing. An agent that can browse can be used for data exfiltration. Follow the chain. ## Handoff cues - State the launch recommendation clearly: go / go with conditions / no-go. - Name the launch blockers and the specific fixes needed to resolve them. - Provide the adversarial test cases so the next operator can re-verify after changes. - Flag any misuse paths that require ongoing monitoring (can't be fully guarded, need alert-based detection). ## Example invocation - Slash: `/ai_guardrail_review <task>` - Natural language: "Use AI Guardrail Review to find the abuse paths before users do." - Example: "Review this agent before we open it to external users — I need to know what could go wrong." - Example: "The agent can now send emails and browse the web — review the guardrails." - Often paired with: `ai_eval_harness`, `ai_agent_spec`, `sec_threat_model`, `safe_high_impact_changes` ## Quality bar - A strong guardrail review finds at least one misuse path that wasn't already obvious. - Every recommended guardrail must have a corresponding adversarial test case. - The launch recommendation must be unambiguous — no "probably fine" hedging. - High-impact paths with no architecture-level guardrail are automatic launch blockers. ## Related workflows - **Pre-launch safety:** `ai_agent_spec` → `ai_guardrail_review` → `ai_eval_harness` (adversarial suite) → `sec_threat_model` (if sensitive data) → launch - **Post-incident hardening:** incident → `safe_live_systems` (contain) → `ai_failure_taxonomy` (classify) → `ai_guardrail_review` (review + harden) → `ai_eval_harness` (re-test) - **Capability expansion:** new tool/capability → `ai_guardrail_review` (assess new abuse surface) → guardrail implementation → `ai_eval_harness` (verify)
