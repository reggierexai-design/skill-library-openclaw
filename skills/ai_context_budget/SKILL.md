--- name: ai_context_budget description: "Shape prompts, memory, and retrieved context to fit the smallest useful budget." user-invocable: true disable-model-invocation: false metadata: {"openclaw":{"emoji":"\ud83e\udde0"}} --- # AI Context Budget ## Purpose - Shape prompts, memory, and retrieved context to fit the smallest useful budget. - Every token in the context window has a cost: inference latency, financial expense, and attention dilution. This skill ensures you're spending those tokens where they earn the most, and cutting them where they don't. ## Use when - Use when an agent is bloated, repetitive, slow, or losing the thread across long runs. - Use when the prompt is approaching or exceeding the model's effective context length. - Use when retrieval results, memory, or tool outputs are crowding out the instructions that actually drive behavior. ## Avoid when - Do not add more context until the required context is clear — cutting noise is more valuable than adding signal. - If the problem is fundamentally a model capability issue (wrong model for the task), no amount of context budgeting will fix it — route to `ai_model_routing` instead. - Do not use this as a substitute for `ai_rag_design` when the core problem is retrieval quality, not retrieval quantity. ## Inputs to gather - Current prompt text (system + user turns) — paste the full thing, not a summary. - Context sources in play: RAG chunks, memory reads, tool outputs, few-shot examples, scratchpad. - Model and its effective context window (not theoretical max — the length where quality degrades). - Observed failure mode: losing the thread, slow inference, degraded instruction-following, cost spikes, or inconsistent behavior. - What the agent must always remember vs. what it can retrieve on demand. ## Operating rules - **Every token should earn its place.** Audit each section of the prompt: does removing it change the agent's behavior on test cases? If not, it's noise. - **Prefer routing and retrieval over giant static prompts.** A 2K system prompt + smart retrieval beats a 10K monolith. Put durable policy in the prompt; put reference material in retrieval. - **Keep durable policy separate from disposable task state.** Policy (rules, constraints, persona) is permanent. State (conversation history, scratchpad, retrieved docs) is ephemeral. Never mix them — when state crowds out policy, the agent goes off-rails. - **Measure before you cut.** Run a baseline eval, make the cut, run the eval again. If performance drops, the context you removed was earning its place. - **Respect the recency bias.** Models attend most to the beginning and end of context. Put critical instructions first, put the current task last, and put reference material in the middle. ## OpenClaw tool pattern - Read the current prompt, tool schema, and runtime constraints before proposing changes. Don't optimize a prompt you haven't read in full. - When proposing cuts, show the before/after diff with token counts for each section. - Test budget changes against `ai_eval_harness` — never assume a cut is safe without evidence. - Track token usage per turn, not just per session. A budget that works on turn 1 may fail on turn 20. ## Expanded workflow 1. **Map the current context.** List every source: system prompt, user message, conversation history, tool outputs, RAG chunks, memory reads, few-shot examples. Measure token count for each. 2. **Classify by role.** Tag each source as: policy (permanent rules), state (ephemeral task data), reference (on-demand lookup material), or padding (legacy content nobody remembers adding). 3. **Cut the padding first.** Remove anything that doesn't change behavior on test cases. Common waste: outdated examples, redundant restatements of rules, "helpful assistant" preamble, context that duplicates what the model already knows. 4. **Compress the policy.** Rewrite rules as concise imperatives. Replace paragraphs with bullet points. Merge overlapping constraints. A 500-word policy section can usually become 150 words without losing any behavioral control. 5. **Gate the reference material.** Move from "always include" to "retrieve when needed." Replace static few-shot examples with a retrieval tool. Replace pasted documentation with RAG. 6. **Design the retrieval contract.** For each gated source, specify: when to retrieve, how many tokens to return, how to rank relevance, and what to do when retrieval fails. 7. **Rebudget and verify.** Run the eval suite before and after. Document what was cut, what was gated, and the measured impact. 8. **End with the new budget.** Show token allocation by section, the retrieval plan, and the measured tradeoffs. ## Output contract - Current context map with token counts per source - Classification of each source: policy / state / reference / padding - Cuts made with before/after token savings - Retrieval contract for gated material - Measured eval impact (baseline vs. new budget) - Remaining risks: what might break at longer conversation lengths, higher retrieval loads, or edge-case inputs ## Failure modes to avoid - **Cutting policy to save tokens.** The worst budget cut is one that removes a constraint the agent was actually following. Always verify with evals. - **Over-gating retrieval.** If the agent can't answer common questions without a retrieval call, latency and cost may increase instead of decreasing. - **Ignoring conversation length effects.** A budget that works on turn 1 will degrade on turn 30 as conversation history grows. Plan for the tail, not the average. - **Optimizing for cost while ignoring quality.** The cheapest prompt is empty. The goal is the smallest budget that preserves the required quality. - **Treating token count as the only metric.** Latency, correctness, and consistency matter too. A 20% token reduction that causes 5% quality loss is usually a bad trade. ## Handoff cues - State the original vs. new token budget and the measured quality impact. - Name the sections that were cut, gated, or compressed — another operator should know what changed. - Flag any context sources that couldn't be safely reduced and need a different approach (e.g., `ai_rag_design` for retrieval quality, `ai_model_routing` for a model with a larger window). - Provide the eval command or test suite so the next operator can re-verify after further changes. ## Example invocation - Slash: `/ai_context_budget <task>` - Natural language: "Use AI Context Budget to trim the prompt down to what actually earns its tokens." - Example: "My agent is losing the thread after 10 turns — help me cut the bloat." - Example: "Audit this prompt and tell me what I can safely remove or gate behind retrieval." - Often paired with: `ai_prompt_system`, `ai_eval_harness`, `ai_rag_design`, `ai_model_routing` ## Quality bar - A strong context budget preserves eval performance while measurably reducing token count. - The output should include before/after token counts and eval scores — no hand-waving. - If you can't measure the impact of a cut, you shouldn't be making it. ## Related workflows - **Prompt optimization cycle:** `ai_context_budget` → `ai_prompt_system` (rewrite) → `ai_eval_harness` (verify) → `ai_context_budget` (re-audit) - **RAG + budget design:** `ai_rag_design` (retrieval) → `ai_context_budget` (gate reference material) → `ai_eval_harness` (end-to-end test) - **Agent scaling:** `ai_agent_spec` (design) → `ai_context_budget` (budget) → `ai_model_routing` (right-size model) → `ai_subagent_decomposition` (split if still too large)
