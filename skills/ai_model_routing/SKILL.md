--- name: ai_model_routing description: "Choose which model handles which work based on quality, speed, cost, and risk." user-invocable: true disable-model-invocation: true metadata: {"openclaw":{"emoji":"\ud83d\udee3\ufe0f"}} --- # AI Model Routing ## Purpose - Choose which model handles which work based on quality, speed, cost, and risk. - Using the most capable model for every request is like using a freight truck for grocery runs — expensive and slow. This skill designs a routing policy that matches task demands to model capabilities, saving cost and latency where possible without silently degrading quality where it matters. ## Use when - Use when a system has multiple models or providers with different strengths, speeds, and costs. - Use when cost or latency is a bottleneck and you suspect over-provisioning (using a bigger model than the task requires). - Use when different task classes clearly need different model capabilities (e.g., creative writing vs. structured extraction vs. safety-critical decisions). ## Avoid when - Do not route randomly or by price alone when task quality matters — the cheapest model that fails the task is the most expensive model. - If you only have one model, there's nothing to route — this skill requires multiple options. - If the task is about designing the agent itself, start with `ai_agent_spec`; routing comes after the spec exists. ## Inputs to gather - Available models with their capabilities, latency profiles, cost per token, and known failure modes. - Task classes the agent handles: creative, analytical, safety-critical, high-volume, low-stakes, tool-use-heavy, etc. - Quality requirements per task class: what's the acceptable accuracy/failure rate? - Cost and latency constraints: what's the budget? What latency SLA exists? - Current routing: is everything going to one model, or is there an existing routing scheme that needs improvement? - Eval data: do you have quality scores per model per task class? If not, that's the first thing to build. ## Operating rules - **Route by task shape, not brand loyalty.** The model that's best for code review may be wrong for creative writing. Match the task's demands (reasoning depth, structured output, safety sensitivity, latency tolerance) to the model's strengths. - **Keep the default path simple.** A routing policy with 20 rules is a maintenance nightmare. Start with 3-5 task classes and expand only when the data shows a gap. - **Define escalation and fallback clearly.** When the primary model fails or is unavailable, where does the request go? What quality degradation is acceptable? What's the retry policy? - **Measure the routing, not just the models.** A routing policy is only as good as its classification accuracy. Track: how often are tasks routed correctly? How often does the fallback fire? What's the cost/quality ratio? - **Protect safety-critical paths.** Safety decisions always get the most capable model, regardless of cost. Never route safety-critical work to a cheaper/faster model without explicit eval evidence that it's safe. ## OpenClaw tool pattern - Read the current routing config and model capabilities before proposing changes. - Use `ai_eval_harness` to build quality baselines per model per task class. Routing decisions must be backed by data, not intuition. - After implementing routing, monitor the distribution: are tasks actually going where expected? Is the fallback firing more than anticipated? - Track cost and latency per task class before and after routing changes. ## Expanded workflow 1. **Inventory task classes and models.** List every distinct task type the agent handles and every available model. Note the quality, speed, and cost characteristics of each model. 2. **Build quality baselines.** For each task class, run evals against each candidate model. Record accuracy, failure rate, and latency. This data drives all routing decisions. 3. **Match tasks to models.** For each task class, assign: a primary model (best quality/cost ratio), a fallback model (when primary is unavailable), and an escalation model (when the task exceeds the primary's capability). 4. **Define classification rules.** How does the system determine which task class a request belongs to? Keyword matching? Embedding similarity? Explicit user selection? The classifier must be simple and testable. 5. **Design the fallback chain.** When the primary model fails: retry with same model? Escalate to stronger model? Return an error? Define the chain for each task class. 6. **Set cost and latency budgets.** Define the acceptable cost per task class and the latency SLA. Verify that the routing policy stays within budget under expected load. 7. **Test the routing end-to-end.** Submit representative requests for each task class and verify they route correctly. Measure cost, latency, and quality. 8. **Document the routing policy.** Compile: task classes, model assignments, classification rules, fallback chains, cost/latency budgets, and monitoring strategy. ## Output contract - Task class inventory with quality/cost/latency requirements per class - Model assignment: primary, fallback, and escalation per task class - Classification rules: how requests get assigned to task classes - Fallback chain: what happens when primary model fails or is unavailable - Cost and latency projections vs. current baseline - Monitoring strategy: what to track, alert thresholds, and review schedule - Eval evidence supporting each routing decision ## Failure modes to avoid - **Routing by price alone.** The cheapest model that fails the task is the most expensive model. Always verify quality before optimizing cost. - **Over-classifying task types.** A routing policy with too many task classes becomes unmaintainable and error-prone. Start simple, expand based on data. - **Silent quality degradation.** If you route to a cheaper model without running evals, you don't know what broke. Always baseline before changing. - **Ignoring the classifier accuracy.** If the task classifier is wrong 30% of the time, your routing policy is wrong 30% of the time. Test the classifier. - **No fallback for model outages.** If the primary model goes down and there's no fallback, the whole system goes down. Every task class needs a fallback. ## Handoff cues - State the routing policy: task classes, model assignments, and classification rules. - Name the eval baselines that support each routing decision. - Flag any task classes where the quality/cost tradeoff is uncertain and needs further eval. - Provide the monitoring dashboard or alert config so the next operator can track routing health. ## Example invocation - Slash: `/ai_model_routing <task>` - Natural language: "Use AI Model Routing to match tasks to the right model without overpaying." - Example: "I have GPT-4, Claude, and a local model — help me route tasks by capability and cost." - Example: "My API bill is too high — which tasks can I safely downgrade to a cheaper model?" - Often paired with: `ai_agent_spec`, `ai_eval_harness`, `ai_context_budget`, `ai_subagent_decomposition` ## Quality bar - Good routing saves measurable cost without silently degrading quality on any task class. - Every routing decision must be backed by eval data — no "I think this model is fine for that task." - The fallback chain must be defined for every task class, and the classifier accuracy must be measured. - If you can't show the before/after cost and quality numbers, the routing isn't done. ## Related workflows - **Agent cost optimization:** `ai_eval_harness` (baseline) → `ai_model_routing` (route) → `ai_eval_harness` (re-verify) → monitor - **Multi-model architecture:** `ai_agent_spec` (design) → `ai_model_routing` (assign models) → `ai_subagent_decomposition` (coordinate) → `ai_eval_harness` (test) - **Performance tuning:** `ai_context_budget` (trim prompts) → `ai_model_routing` (right-size model) → `ai_eval_harness` (verify quality preserved)
