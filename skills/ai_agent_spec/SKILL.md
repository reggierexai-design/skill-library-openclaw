--- name: ai_agent_spec description: "Specify an agent workflow, boundaries, tools, and outputs before implementation." user-invocable: true disable-model-invocation: false metadata: {"openclaw":{"emoji":"\ud83e\udd16"}} --- # AI Agent Spec ## Purpose - Specify an agent workflow, boundaries, tools, and outputs before implementation. - Produces a concrete, testable design document that bridges the gap between "I want an agent that does X" and "here is exactly how it works, what can go wrong, and how we know it's working." ## Use when - Use when designing a new agent, assistant, or semi-automated workflow. - Use when an existing agent needs restructuring — new tools, changed boundaries, or better failure handling. - Use when the team is about to start implementation and the spec is still vague or implicit. ## Avoid when - Do not jump into implementation when the job, tools, or failure modes are still undefined — spec first, code second. - Do not use this for simple prompt tweaks; use `ai_prompt_system` instead. - If the task is primarily about evaluation, route to `ai_eval_harness` first. ## Inputs to gather - The agent's primary job: what decision or action it owns end-to-end. - Target users and their mental model: do they trust the agent, fear it, or ignore it? - Available models, tools, APIs, and context sources (RAG, memory, databases). - Failure tolerance: what happens if the agent is wrong 1% of the time? 10%? 50%? - Existing prompts, evals, logs, incidents, or product requirements that constrain the design. - Handoff targets: who or what receives the agent's output, and what format do they need? ## Operating rules - **Define the job before the prompt.** An agent without a clear job becomes a chatbot with extra steps. Write the job as a single sentence: "This agent [does X] for [user Y] so that [outcome Z]." - **Map the decision loop explicitly.** Every agent follows: observe → reason → act → verify. Name what triggers each step, what information flows between them, and where human approval gates exist. - **Design for the worst case, not the happy path.** List every plausible failure mode (bad input, tool outage, model hallucination, context overflow) and specify the fallback for each. An agent without fallbacks is an agent that will embarrass you. - **Make tool use intentional.** Every tool in the agent's kit should map to a specific job it alone can do. Remove tools that overlap or that the agent uses for convenience rather than necessity — more tools means more failure surface. - **Spec the memory contract.** What does the agent remember between turns? Between sessions? What must it forget? Unconstrained memory is a bug farm; unconstrained forgetting is a support burden. ## OpenClaw tool pattern - Read the current prompt, tool schema, examples, and runtime constraints before proposing architecture changes. Ground the spec in what exists, not in an idealized version. - Treat agent behavior as a system: prompts, tools, context, memory, evals, and failure handling must agree. A great prompt with the wrong tools or no evals will fail in production. - Prefer designs that are testable, observable, and cheap to iterate. A spec that requires a full deployment to validate is a spec that won't get validated. - When proposing changes, show the before/after diff for prompts, tool configs, and guardrail rules — not just a description. ## Expanded workflow 1. **Define the agent goal and user.** Write the job sentence. Name the primary user, the decision the agent owns, and the outcome it produces. 2. **Map the decision loop.** Draw the observe → reason → act → verify cycle. Identify where the agent reads context, where it reasons, where it calls tools, and where it checks its own output. 3. **Specify tool access and permissions.** List every tool, why it's needed, what it can do, and what guardrails constrain it. If a tool can write to production, name the approval gate. 4. **Define constraints, permissions, and failure handling.** For each failure mode, specify: detection signal, fallback behavior, and escalation path. 5. **Design the eval strategy.** Define at least 3 test cases: one happy path, one edge case, one adversarial input. Name the metric that determines pass/fail. 6. **Write the memory policy.** What persists between turns? Between sessions? What is ephemeral? What must be explicitly cleared? 7. **Produce the agent spec document.** Compile into a single artifact: goal, decision loop, tools, constraints, evals, memory, failure handling, and implementation order. 8. **Review against the original request.** Remove anything that doesn't change the outcome. End with the exact next action: implementation, review, or de-risk experiment. ## Output contract - Agent job sentence (who, what, why) - Decision loop diagram or description - Tool manifest with permissions and guardrails - Failure mode table: failure → detection → fallback → escalation - Eval strategy with test cases and pass/fail metrics - Memory policy: what persists, what's ephemeral, what's cleared - Implementation order: what to build first, what depends on what - Open questions and the smallest next experiment to de-risk the design ## Failure modes to avoid - **Solving a systems problem with prompt wording alone.** If the agent is unreliable, the fix is usually architecture (tools, memory, guardrails), not a longer prompt. - **Adding retrieval, memory, or tools without a concrete failure mode they solve.** Every capability adds failure surface. If you can't name the failure it prevents, don't add it. - **Skipping eval design.** An agent without evals is an agent you can't improve, because you can't tell whether a change helped or hurt. - **Specifying behavior without boundaries.** An agent that can do anything will do the wrong thing at the worst time. Boundaries are the most important part of the spec. - **Optimistic concurrency assumptions.** Assuming the agent will always get clean input, fast tool responses, and consistent context is assuming away the hard parts. ## Handoff cues - State the spec status: draft, reviewed, approved, or needs specific input. - Name the first implementation task and who should own it. - Flag any eval infrastructure that needs to be built before the agent can be tested. - List any model, tool, or access dependencies that block implementation. ## Example invocation - Slash: `/ai_agent_spec <task>` - Natural language: "Use AI Agent Spec to design the agent before we start building." - Example: "Design the agent path so it is testable, safe, and not dependent on luck." - Example: "Review this prompt/tool system and tell me the real failure modes and fixes." - Often paired with: `ai_prompt_system`, `ai_eval_harness`, `ai_guardrail_review` ## Quality bar - A good agent spec makes vague or unsafe behavior **hard by design** — not just warned against, but structurally prevented. - Another operator should be able to implement the agent from the spec without asking clarifying questions. - The spec must include at least one failure mode per tool and one adversarial test case — if it doesn't, it's not done. ## Related workflows - **Full agent lifecycle:** `ai_agent_spec` → `ai_prompt_system` → `ai_tool_schema_review` → `ai_eval_harness` → `ai_guardrail_review` → `eng_test_strategy` - **Agent debugging:** `eng_debug_systematically` → `ai_failure_taxonomy` → `ai_agent_spec` (redesign) → `ai_eval_harness` (re-validate) - **Multi-agent orchestration:** `ai_agent_spec` (per agent) → `ai_subagent_decomposition` (orchestration) → `ai_model_routing` (model assignment) - **Safety review:** `ai_agent_spec` → `ai_guardrail_review` → `sec_threat_model` → `safe_high_impact_changes`
