--- name: ai_tool_schema_review description: "Review tool names, arguments, and failure contracts so agents can use them reliably." user-invocable: true disable-model-invocation: true metadata: {"openclaw":{"emoji":"\ud83e\uddf0"}} --- # AI Tool Schema Review ## Purpose - Review tool names, arguments, and failure contracts so agents can use them reliably. - When an agent picks the wrong tool, passes bad arguments, or misinterprets the result, the root cause is often the tool schema, not the prompt. This skill audits the tool surface for the patterns that confuse models: ambiguous names, overlapping tools, missing error contracts, and arguments that are easy to misuse. ## Use when - Use when tools exist but the agent struggles to pick or call them correctly. - Use when adding a new tool — review the schema before the agent ever sees it. - Use when the agent is calling the wrong tool, omitting required arguments, or hallucinating optional ones. ## Avoid when - Do not blame the model before checking whether the tool surface is coherent — most "model confusion" is actually "schema confusion." - If the problem is the agent's decision logic (when to use tools vs. when to answer directly), use `ai_prompt_system` instead. - If the tool itself is broken (returns wrong data), use `eng_bug_triage` instead. ## Inputs to gather - The full tool manifest: every tool the agent can access, with name, description, parameters, and return schema. - Tool call logs: recent calls, including misfires (wrong tool, wrong args, ignored errors). - The agent's prompt section on tool usage — does it explain when and how to use each tool? - Known failure patterns: which tools does the agent misuse most? What are the common argument mistakes? ## Operating rules - **Tools should advertise intent, not implementation trivia.** A tool named `database_query` tells the model nothing about when to use it. A tool named `search_customer_by_email` tells the model exactly what it's for. Name tools by the job they do for the agent, not by the API they wrap. - **Arguments should be minimal, typed, and hard to misuse.** Every required argument is a potential failure point. Remove arguments the agent can't meaningfully set. Use enums instead of free-text where possible. Add validation hints in the description. - **Failure modes should be explicit and recoverable.** Every tool should document: what it returns on success, what it returns on error, and what the agent should do when it gets an error. An undocumented error path is an unhandled error path. - **Overlapping tools must be distinguished.** If two tools can do similar things, the agent will pick randomly. Either merge them, rename them to clarify the distinction, or add a routing rule that decides which one to use. - **Examples belong in the schema.** A tool description with 2-3 example calls teaches the model more than a paragraph of prose. Show the happy path and one error case. ## OpenClaw tool pattern - Read the current tool schemas and call logs before proposing changes. You can't review what you haven't read. - After schema changes, test the agent's tool selection accuracy with `ai_eval_harness`. - Track tool call success rates before and after changes — the metric that matters is "does the agent pick the right tool with the right args?" - If schema changes affect safety-critical tools, coordinate with `ai_guardrail_review`. ## Expanded workflow 1. **Inventory the tool set.** List every tool with its name, description, parameters, and return schema. Flag any tools that overlap in purpose. 2. **Audit naming.** For each tool: does the name tell the model what the tool does and when to use it? Flag names that are implementation-focused (e.g., `api_v2_call`) instead of intent-focused (e.g., `send_email_to_user`). 3. **Audit arguments.** For each tool: are there arguments the agent can't meaningfully set? Are there free-text fields that should be enums? Are required vs. optional clearly distinguished? Flag arguments that are easy to misuse. 4. **Audit error contracts.** For each tool: what does it return on success? On error? Does the description tell the agent what to do with errors? Flag any tool without documented error handling. 5. **Check for overlaps.** If two tools can serve the same purpose, flag the overlap. Propose: merge, rename to distinguish, or add a routing rule. 6. **Add examples.** For each tool, write 2-3 example calls: one happy path, one edge case, one error case. Add these to the tool description. 7. **Rank by impact.** Sort the findings by how often the agent misuses each tool. Fix the highest-impact problems first. 8. **Propose the fixes.** For each finding, write the specific schema change. Show before/after. Estimate the impact on agent accuracy. ## Output contract - Tool inventory with naming, argument, and error contract assessment per tool - Naming fixes: before/after for each renamed tool - Argument fixes: removed, retyped, or constrained arguments per tool - Error contract additions: documented success/error/behavior per tool - Overlap resolution: merged, renamed, or routed tools - Example calls added to each tool description - Impact ranking: highest-leverage fixes first ## Failure modes to avoid - **Renaming without testing.** A name that seems clearer to you might not be clearer to the model. Always test naming changes with `ai_eval_harness`. - **Over-constraining arguments.** Enums are great for preventing errors, but an enum that's too narrow will cause the agent to fail on valid inputs. Leave escape hatches where they matter. - **Adding tools without removing overlaps.** More tools means more decision surface and more confusion. Before adding a tool, check if an existing one can do the job with a schema tweak. - **Documenting error paths that don't exist.** If the tool doesn't actually return the errors you documented, the agent will handle them incorrectly. Verify error contracts against the real implementation. - **Ignoring the agent's actual usage patterns.** If the agent never calls a particular tool, the schema review is academic. Focus on tools the agent actually uses and misuses. ## Handoff cues - State the tool schema changes and the expected impact on agent accuracy. - Name the eval cases that should be run to verify the improvements. - Flag any tools where the error contract couldn't be verified against the real implementation. - Note any tools that should be removed entirely (unused or redundant) and the migration path for existing prompts. ## Example invocation - Slash: `/ai_tool_schema_review <task>` - Natural language: "Use AI Tool Schema Review to fix tool schemas so the agent stops misfiring." - Example: "The agent keeps calling the wrong tool — review the schemas and fix the confusion." - Example: "Adding a new tool — review the schema before the agent ever sees it." - Often paired with: `ai_agent_spec`, `ai_prompt_system`, `ai_eval_harness`, `ai_guardrail_review` ## Quality bar - A good tool surface lets the model choose the right action with less guesswork. - After the review, tool call accuracy should measurably improve on the eval suite. - Every tool must have documented error contracts — no "it might fail somehow" gaps. - Overlapping tools must be resolved: merged, renamed, or routed — no ambiguities left standing. ## Related workflows - **Tool surface design:** `ai_agent_spec` (design agent) → `ai_tool_schema_review` (design tool surface) → `ai_prompt_system` (tool usage instructions) → `ai_eval_harness` (test tool accuracy) - **Tool debugging:** `ai_failure_taxonomy` (classify tool errors) → `ai_tool_schema_review` (fix schemas) → `ai_eval_harness` (verify) - **Safety tooling:** `ai_guardrail_review` (find abuse paths) → `ai_tool_schema_review` (add safety constraints to tool schemas) → `ai_eval_harness` (test adversarial cases)
