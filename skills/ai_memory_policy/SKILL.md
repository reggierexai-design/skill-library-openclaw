# AI Memory Policy ## Purpose - Define what an agent should remember, forget, summarize, or ask again over time. - Unconstrained memory is a bug farm: stale facts, leaked context between users, and ever-growing state that degrades the model's attention. This skill designs a memory contract that keeps the agent sharp and safe across sessions. ## Use when - Use when an agent will run repeatedly and needs stable memory boundaries. - Use when the agent is accumulating context that degrades performance over time. - Use when different users or sessions need memory isolation (no cross-contamination). ## Avoid when - Do not use for one-shot tasks with no ongoing context — memory policy adds complexity that's only justified by repeated execution. - If the problem is prompt bloat, route to `ai_context_budget` instead. - If the problem is retrieval quality, route to `ai_rag_design` instead. ## Inputs to gather - The agent's job and its memory-dependent decisions: which decisions require remembering past interactions? - Memory types in play: conversation history, user preferences, accumulated knowledge, session state, long-term facts. - User boundaries: is memory per-user, per-session, per-conversation, or shared? - Freshness requirements: how quickly does stored information become stale? What's the cost of acting on stale information? - Privacy constraints: PII handling, data retention limits, cross-user isolation requirements. - Current memory implementation: how is memory stored, retrieved, and cleared today? ## Operating rules - **Distinguish durable from disposable memory.** Durable: user preferences, learned facts, standing constraints. Disposable: conversation history, scratchpad state, temporary context. Never mix them — disposable memory should be summarizable and discardable. - **Design for the forgetting curve.** Information has a half-life. User preferences are durable; conversation context from 3 days ago is not. Define retention windows per memory type and enforce them. - **Isolate by boundary.** Memory that crosses user boundaries is a privacy bug. Memory that crosses session boundaries when it shouldn't is a coherence bug. Define the isolation boundary and enforce it architecturally, not just in the prompt. - **Summarize before storing.** Raw conversation history is expensive and noisy. Summarize to extract decisions, preferences, and facts before persisting. The summary is the memory; the raw transcript is the source. - **Make forgetting safe.** When memory is cleared or summarized, the agent should gracefully re-ask rather than hallucinate. Define what happens when the agent doesn't know: ask the user, use a safe default, or acknowledge the gap. ## OpenClaw tool pattern - Read the current memory implementation (storage, retrieval, retention) before proposing changes. You can't design a policy without understanding the existing system. - Test memory behavior across session boundaries: start a conversation, end it, start a new one, and verify that the agent remembers what it should and forgets what it shouldn't. - After implementing the memory policy, add test cases to `ai_eval_harness` that verify isolation, retention, and graceful forgetting. - When memory affects safety (e.g., remembering PII), coordinate with `ai_guardrail_review` to ensure the policy meets privacy requirements. ## Expanded workflow 1. **Inventory the agent's memory-dependent decisions.** List every decision the agent makes that depends on remembering past interactions. For each, note: what must be remembered, how long it's valid, and what happens if it's forgotten. 2. **Classify memory by type.** Tag each memory as: durable (preferences, facts), semi-durable (learned patterns, recent context), or disposable (conversation scratchpad, temporary state). 3. **Define retention policies.** For each type: how long is it retained? When is it summarized vs. discarded? What triggers a refresh (new information, time elapsed, session change)? 4. **Define isolation boundaries.** Per-user? Per-session? Per-conversation? Shared? Write the boundary rule and verify there are no cross-boundary leaks. 5. **Design the summarization strategy.** For semi-durable and disposable memory: what gets extracted, what format the summary takes, and when summarization occurs. 6. **Define the forgetting behavior.** When memory is cleared or expired: does the agent re-ask, use a default, or acknowledge the gap? Write the behavior for each case. 7. **Test across boundaries.** Write test cases for: session restart, user switch, conversation handoff, memory expiry. Verify isolation and graceful degradation. 8. **Document the memory contract.** Compile: memory types, retention policies, isolation boundaries, summarization strategy, forgetting behavior, and test cases. ## Output contract - Memory type inventory with retention and isolation rules per type - Summarization strategy: what's extracted, when, and in what format - Forgetting behavior: how the agent handles expired or missing memory - Isolation boundary definition with no-cross-contamination guarantee - Test cases for session restart, user switch, and memory expiry - Privacy and PII handling rules if applicable ## Failure modes to avoid - **Remembering everything forever.** Unbounded memory accumulates noise, degrades performance, and creates privacy risk. Always define retention limits. - **Cross-contaminating user memory.** If user A's preferences leak into user B's session, that's a privacy violation and a coherence bug. Isolate architectically. - **Summarizing away critical details.** A summary that drops a key constraint or preference is worse than no memory at all. Test summaries against the original to verify fidelity. - **Silent hallucination when memory is missing.** If the agent doesn't remember and doesn't re-ask, it will fabricate. The forgetting behavior must be explicit. - **Over-engineering for a one-shot agent.** If the agent runs once and exits, it doesn't need a memory policy. Don't add complexity that isn't justified. ## Handoff cues - State the memory contract: types, retention, isolation, and forgetting behavior. - Name the implementation changes needed (storage schema, retrieval logic, retention enforcement). - Flag any privacy implications that need review (`ai_guardrail_review`, `safe_pii_minimization`). - Provide the test cases so the next operator can verify the policy works correctly. ## Example invocation - Slash: `/ai_memory_policy <task>` - Natural language: "Use AI Memory Policy to define what the agent should remember, forget, and re-ask." - Example: "The agent is getting confused across sessions — help me design a memory contract." - Example: "Users are seeing each other's context — fix the memory isolation." - Often paired with: `ai_agent_spec`, `ai_context_budget`, `ai_eval_harness`, `safe_pii_minimization` ## Quality bar - A good memory policy prevents cross-contamination, limits stale context, and ensures graceful forgetting. - The isolation boundary must be enforced architectically, not just stated in the prompt. - Every memory type must have a defined retention limit — no "remember forever" defaults. - The output must include test cases for session restart and user isolation. ## Related workflows - **Agent design:** `ai_agent_spec` → `ai_memory_policy` (define contract) → `ai_context_budget` (fit to window) → `ai_eval_harness` (test memory behavior) - **Privacy audit:** `ai_memory_policy` → `safe_pii_minimization` (PII handling) → `ai_guardrail_review` (safety) → `sec_data_flow_review` (data flow) - **Performance tuning:** `ai_context_budget` (token audit) → `ai_memory_policy` (summarization) → `ai_rag_design` (offload to retrieval) → `ai_eval_harness` (re-test)
