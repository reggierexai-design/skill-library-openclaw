--- name: ai_prompt_system description: "Turn scattered instructions into a stable prompt architecture with clear sections and duties." user-invocable: true disable-model-invocation: false metadata: {"openclaw":{"emoji":"\ud83d\udcdd"}} --- # AI Prompt System ## Purpose - Turn scattered instructions into a stable prompt architecture with clear sections and duties. - A prompt that grew organically is a prompt that's fragile: instructions conflict, priorities are implicit, and nobody can tell what the model will actually do. This skill replaces "instructions that worked once" with a prompt architecture that works reliably and can be maintained by anyone. ## Use when - Use when prompts are long, inconsistent, or patched together from many experiments. - Use when multiple people edit the same prompt and it's drifting toward incoherence. - Use when the prompt is approaching the context window limit and needs structural pruning, not just trimming. ## Avoid when - Do not polish wording before clarifying the real operating rules — structure before prose. - If the task is a small tweak to an existing well-structured prompt, use `ai_prompt_regression_review` instead. - If the problem is context budget, use `ai_context_budget` first — restructure after trimming. ## Inputs to gather - The full current prompt text — not a summary, not "it's about 2 pages." Paste the whole thing. - Known behavioral issues: what does the agent do wrong? What edge cases break? What safety rules does it ignore? - The agent's tools, memory, and context sources — the prompt must agree with the system. - Who maintains this prompt? How often does it change? What's the review process? - Eval data: any test cases or quality measurements that define "working correctly." ## Operating rules - **Structure beats clever phrasing.** A well-organized prompt with plain language outperforms a beautifully written prompt where instructions conflict. The model follows structure more reliably than rhetoric. - **Separate policy, process, style, and tool guidance.** Policy: what must always happen. Process: the step-by-step workflow. Style: tone, format, length. Tools: when and how to use each tool. Never mix them — when policy and style share a section, the model can't tell which takes priority. - **Write prompts for maintainers, not just for the current run.** Every section should have a clear purpose that another editor can understand. If you can't explain why a section exists, it probably shouldn't. - **Put the most important instructions first and last.** Models attend most to the beginning and end of the prompt. Critical constraints go at the top. Critical reminders go at the bottom. Context and reference material go in the middle. - **One instruction per line.** Compound instructions ("Be concise and always cite sources and never use jargon") are ambiguous when they conflict. Split them: one rule for conciseness, one for citations, one for jargon. Then prioritize. ## OpenClaw tool pattern - Read the current prompt, tool schema, and runtime config in full before proposing changes. - After restructuring, run `ai_eval_harness` to verify behavior is preserved — restructure should not regress. - Use `ai_prompt_regression_review` if the change is to an existing production prompt — always diff and test. - Store prompt versions in the workspace with clear naming (e.g., `prompts/v2-system.md`) so changes are traceable. ## Expanded workflow 1. **Audit the current prompt.** Paste the full text. Identify every instruction, constraint, style guide, and context source. Flag duplicates, conflicts, and implicit priorities. 2. **Classify each instruction.** Tag as: policy (always enforce), process (follow this workflow), style (format/tone), tool guidance (use this tool when), or padding (legacy content nobody remembers adding). 3. **Remove padding and duplicates.** Cut anything that doesn't change behavior on test cases. Merge duplicate instructions. Resolve conflicts by explicit priority ranking. 4. **Reorganize into sections.** Standard structure: (1) Role & Purpose, (2) Hard Constraints / Policy, (3) Workflow / Process, (4) Tool Usage Rules, (5) Output Format / Style, (6) Edge Case Handling, (7) Reminders / Priority Order. 5. **Rewrite each section for clarity.** One instruction per line. Plain language. No hedging. Every rule should be unambiguously testable: "Does this instruction change the agent's behavior on a specific input?" If you can't tell, the instruction is too vague. 6. **Set explicit priorities.** When rules conflict, which wins? Write the priority order explicitly. "Safety > accuracy > helpfulness > conciseness" is better than hoping the model figures it out. 7. **Add the version header.** Include: version number, last modified date, owner, and a one-line change summary. Future editors need to know what changed and when. 8. **Test against the eval suite.** Run the restructured prompt against existing test cases. If behavior regressed, the restructure introduced a conflict — find and fix it. ## Output contract - Full restructured prompt text, ready to use - Section map: purpose of each section - Removed items: what was cut and why - Priority ranking: which rules override which - Change summary: what's different from the original and why - Eval comparison: behavior before and after restructure ## Failure modes to avoid - **Solving a systems problem with prompt wording.** If the agent misuses tools, the fix is usually in the tool schema or architecture, not in adding "be careful with tools" to the prompt. - **Preserving instructions that don't change behavior.** If removing an instruction doesn't change eval results, it was noise. Don't keep it out of sentimentality. - **Implicit priority ordering.** "Safety first" means nothing if the model also has instructions to "be helpful" and "be concise" without a clear ranking. Explicit priority prevents the model from guessing. - **Over-abstracting.** A prompt that says "follow best practices" is useless. A prompt that says "always verify tool outputs before acting on them" is actionable. Prefer concrete instructions over abstract principles. - **Skipping the eval.** Restructuring always risks behavioral regression. If you don't test, you don't know what broke. ## Handoff cues - State what was restructured and what was preserved — another operator needs to know the scope of change. - Provide the before/after diff and the eval comparison. - Flag any sections where the priority ordering is still ambiguous and needs human judgment. - Name the prompt version and storage location so future edits are traceable. ## Example invocation - Slash: `/ai_prompt_system <task>` - Natural language: "Use AI Prompt System to restructure this prompt so it's maintainable and reliable." - Example: "This prompt is a mess of patches — restructure it into something coherent." - Example: "Multiple people edit this prompt and it's drifting — give it a stable architecture." - Often paired with: `ai_eval_harness`, `ai_prompt_regression_review`, `ai_context_budget`, `ai_guardrail_review` ## Quality bar - A good prompt system is easy to reason about and hard to accidentally break. - Another operator should be able to read the prompt and understand every instruction's purpose without asking. - The restructured prompt must pass the existing eval suite with no regressions. - If you can't explain why a section exists, it's not done — cut it or clarify it. ## Related workflows - **Prompt build cycle:** `ai_prompt_system` (structure) → `ai_eval_harness` (baseline) → `ai_context_budget` (trim) → `ai_prompt_regression_review` (verify changes) - **Safety hardening:** `ai_guardrail_review` (find gaps) → `ai_prompt_system` (add constraints) → `ai_eval_harness` (test safety cases) - **Agent design:** `ai_agent_spec` (design) → `ai_prompt_system` (implement) → `ai_tool_schema_review` (tool surface) → `ai_eval_harness` (verify)
