--- name: ai_rag_design description: "Design retrieval, chunking, freshness, and citation behavior for grounded question answering." user-invocable: true disable-model-invocation: true metadata: {"openclaw":{"emoji":"\ud83d\udcda"}} --- # AI RAG Design ## Purpose - Design retrieval, chunking, freshness, and citation behavior for grounded question answering. - RAG without design is just "stuff documents into a vector store and hope." This skill makes retrieval intentional: the right chunks, the right freshness, the right citation behavior, and the right fallback when retrieval fails. ## Use when - Use when answers must rely on evolving source material instead of model memory alone. - Use when the current RAG system returns irrelevant, stale, or ungrounded results. - Use when building a new knowledge-grounded agent from scratch. ## Avoid when - Do not build retrieval before clarifying the corpus, users, and freshness needs — RAG without requirements is expensive guesswork. - If the problem is prompt quality, not retrieval quality, use `ai_prompt_system` instead. - If the problem is context window budget, use `ai_context_budget` first — RAG may not be the bottleneck. ## Inputs to gather - Source corpus: what documents, how large, how often updated, what formats? - User question types: factual lookup, synthesis across documents, comparison, how-to guidance, or analytical reasoning? - Freshness requirements: does the answer need to reflect updates from the last hour? Last day? Last month? - Citation expectations: must the agent cite specific sources? Link to documents? Quote passages? - Current retrieval system: embedding model, chunk size, retrieval method (vector, keyword, hybrid), top-k setting. - Known failure modes: irrelevant chunks, stale information, missing documents, hallucinated citations. ## Operating rules - **Retrieval quality beats vector count.** More chunks in the index doesn't mean better answers. If the top-k results are irrelevant, adding more vectors just adds noise. Optimize retrieval precision before expanding the corpus. - **Design for freshness, provenance, and fallback behavior.** Freshness: when was this chunk last verified? Provenance: where did it come from? Fallback: what does the agent say when retrieval returns nothing useful? Without these three, RAG is a citation fabrication engine. - **Chunk for answerability, not just token size.** A 512-token chunk that splits a table from its header is useless. A 200-token chunk that captures a complete Q&A pair is valuable. Chunk boundaries should respect document structure, not just character count. - **Always define the "no good retrieval" behavior.** The worst RAG failure isn't returning bad chunks — it's the agent confidently answering from bad chunks. When retrieval quality is low, the agent should say "I don't have good information on that" rather than hallucinating from noise. - **Test retrieval independently from generation.** If the retrieved chunks are wrong, the answer will be wrong regardless of how good the prompt is. Evaluate retrieval precision and recall separately. ## OpenClaw tool pattern - Read the current retrieval config (embedding model, chunk size, top-k, reranking) before proposing changes. - Test retrieval quality independently: given a question, are the top-k chunks actually relevant? Measure this before tuning the generation prompt. - After implementing RAG changes, add retrieval eval cases to `ai_eval_harness` — both chunk-level (is the right chunk retrieved?) and answer-level (is the final answer grounded?). - When freshness matters, design a re-indexing pipeline, not just a one-time load. ## Expanded workflow 1. **Map question types and source corpus.** List the distinct question categories users ask. For each, identify: what documents contain the answer, how fresh the answer must be, and what citation format is expected. 2. **Design the chunking strategy.** For each document type, define: chunk size (respect structure, not just tokens), chunk overlap (to preserve cross-boundary context), metadata to attach (source, date, section heading, version), and chunk boundaries (respect paragraphs, tables, code blocks, not arbitrary character counts). 3. **Design the retrieval pipeline.** Choose: embedding model (quality vs. cost), retrieval method (vector, keyword/BM25, hybrid), top-k (start with 5, adjust based on eval), reranking (if needed for precision), and filtering (by date, source, category). 4. **Design the freshness policy.** Define: how often the index is updated, what triggers a re-index (new document, edited document, time-based), how stale documents are detected, and how the agent signals "this information may be outdated." 5. **Design citation behavior.** Specify: does the agent cite sources inline? List references at the end? Link to source documents? Quote relevant passages? What happens when multiple sources conflict? 6. **Design the fallback behavior.** When retrieval returns no relevant results: does the agent say "I don't know"? Ask the user to rephrase? Fall back to model knowledge (with a disclaimer)? Define this explicitly. 7. **Build retrieval eval cases.** For each question type, create: a question, the expected source chunks, and the expected answer characteristics. This becomes the retrieval test suite. 8. **Document the RAG design.** Compile: corpus, chunking, retrieval, freshness, citation, fallback, eval cases, and known gaps. ## Output contract - Question type inventory with freshness and citation requirements per type - Chunking strategy: size, overlap, boundaries, and metadata per document type - Retrieval pipeline: embedding model, method, top-k, reranking, filtering - Freshness policy: re-index frequency, staleness detection, outdated signal - Citation behavior: format, conflict handling, and source linking - Fallback behavior: what the agent does when retrieval fails - Retrieval eval cases: questions, expected chunks, expected answer characteristics - Known gaps: question types not yet covered, document types not yet chunked ## Failure modes to avoid - **Chunking by token count alone.** Arbitrary chunk boundaries destroy document structure. A table split from its header, a procedure split from its prerequisites, a code block split from its explanation — all produce useless chunks. - **Ignoring freshness.** Stale RAG is worse than no RAG — the agent confidently cites outdated information. If the corpus changes, freshness must be designed, not assumed. - **No fallback for empty retrieval.** When the top-k chunks are all irrelevant, the model will hallucinate an answer from whatever noise it retrieved. Define the "no good results" behavior explicitly. - **Evaluating only the final answer.** If the answer is wrong because retrieval was bad, no amount of prompt tuning will fix it. Evaluate retrieval independently. - **Over-indexing.** Adding every document to the index increases noise. Only index documents that are likely to be queried. A focused corpus outperforms a comprehensive one. ## Handoff cues - State the RAG design: chunking, retrieval, freshness, citation, and fallback — another operator needs the full picture. - Name the retrieval eval cases so the next operator can re-test after changes. - Flag any question types where retrieval quality is uncertain and needs further tuning. - Note the embedding model and index size for cost and latency planning. ## Example invocation - Slash: `/ai_rag_design <task>` - Natural language: "Use AI RAG Design to build a retrieval system that actually returns the right chunks." - Example: "My agent keeps citing stale docs — help me design freshness and citation behavior." - Example: "Building a new knowledge base — design the chunking and retrieval pipeline before I start." - Often paired with: `ai_context_budget`, `ai_prompt_system`, `ai_eval_harness`, `ai_memory_policy` ## Quality bar - A strong RAG design improves trust, not just recall — the agent should be right more often AND say "I don't know" when it's not sure. - Retrieval precision on the eval cases must be measurable: what fraction of top-k chunks are actually relevant? - The freshness policy must be concrete: re-index frequency, staleness detection method, outdated signal. - The fallback behavior must be defined: what the agent says when retrieval returns nothing useful. ## Related workflows - **RAG build cycle:** `ai_rag_design` (design) → implement chunking/indexing → `ai_eval_harness` (retrieval eval) → `ai_prompt_system` (generation prompt) → `ai_eval_harness` (end-to-end eval) - **RAG quality fix:** `ai_failure_taxonomy` (classify retrieval failures) → `ai_rag_design` (redesign) → `ai_eval_harness` (re-test) - **Context budget + RAG:** `ai_context_budget` (audit) → `ai_rag_design` (offload static content to retrieval) → `ai_eval_harness` (verify no regressions)
