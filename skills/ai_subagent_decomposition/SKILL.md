--- name: ai_subagent_decomposition description: "Split a complex job into safe subagent tracks with bounded scope, interfaces, and merge points." user-invocable: true disable-model-invocation: false metadata: {"openclaw":{"emoji":"\ud83e\ude93"}} --- # AI Subagent Decomposition ## Purpose - Split a complex job into safe subagent tracks with bounded scope, interfaces, and merge points. - One agent trying to do everything is one agent that does nothing well. This skill decomposes complex workflows into focused subagents, each with a clear job, bounded scope, and defined interfaces — then designs the orchestration layer that coordinates them. ## Use when - Use when a task is large enough to benefit from parallel or delegated execution. - Use when a single agent's context window or capability can't cover the full job. - Use when different parts of the job need different models (e.g., creative vs. analytical vs. safety-critical). ## Avoid when - Do not decompose when coordination cost would exceed the work itself — a simple task done by one agent beats a complex orchestration for a trivial job. - If the task can be handled by a single well-prompted agent, decomposition is over-engineering. - Do not decompose to avoid writing a good prompt — decomposition is architecture, not procrastination. ## Inputs to gather - The full job: what needs to happen end-to-end, from input to final output. - Constraint surface: what must be sequential (dependencies), what can be parallel, what needs human approval. - Model and capability requirements: does any sub-task need a stronger model? A faster model? A model with specific tool access? - Data flow: what information passes between sub-tasks? What's the interface format? - Failure isolation: if one subagent fails, does it kill the whole job or can the others continue? - Current architecture: is there an existing single-agent design that needs splitting? ## Operating rules - **Decompose by job, not by step.** A subagent should own a complete job ("research the market") not a micro-step ("search for competitor names"). Steps without jobs create orchestration nightmares. - **Define the interface before building the agent.** What does each subagent receive? What does it produce? What format? What's the schema? If you can't write the interface, you can't write the agent. - **Every subagent gets bounded scope.** Each subagent should have: a clear job sentence, explicit inputs, a defined output format, and stated boundaries (what it does NOT do). A subagent without boundaries will expand to fill the context window and forget its actual job. - **Design the merge points explicitly.** Where do subagent outputs combine? Who resolves conflicts? What happens when subagents disagree? Merge is the hardest part of decomposition — design it before you need it. - **Keep the orchestration layer thin.** The orchestrator should route, merge, and escalate — not reason about the domain. A fat orchestrator is a single-agent design in disguise. ## OpenClaw tool pattern - Read the current agent spec and workflow before decomposing. Understand what the single agent is doing before splitting it up. - Design each subagent's spec using `ai_agent_spec`, then route models using `ai_model_routing`. - Test each subagent independently with `ai_eval_harness`, then test the full orchestration end-to-end. - After decomposition, verify that the total cost (multiple model calls + orchestration overhead) doesn't exceed the single-agent approach. ## Expanded workflow 1. **Define the full job end-to-end.** Write the complete workflow from input to final output. Identify every distinct job within it. 2. **Identify decomposition boundaries.** Where are natural seams? Look for: different capabilities needed (research vs. writing vs. review), different data dependencies (parallel vs. sequential), different failure modes (safety-critical vs. best-effort), and different model requirements (fast/cheap vs. strong/expensive). 3. **Define subagent jobs.** For each subagent, write: job sentence, inputs, output format, boundaries, and failure handling. One job per subagent — no multitasking. 4. **Design the interfaces.** For each handoff between subagents, define: the data schema, the format, the validation rules, and what happens on invalid input. 5. **Design the merge points.** For each point where subagent outputs combine, define: who merges, how conflicts are resolved, what the merge output looks like, and what happens if a subagent fails to deliver. 6. **Design the orchestration layer.** Define: routing logic (which subagent handles what), sequencing (what runs in parallel vs. sequential), error handling (what happens when a subagent fails), and human approval gates. 7. **Assign models per subagent.** Use `ai_model_routing` to match each subagent's capability needs to the right model. Not every subagent needs the most expensive model. 8. **Test independently, then together.** Evaluate each subagent with `ai_eval_harness` first, then test the full orchestration end-to-end. 9. **Document the decomposition.** Compile: subagent specs, interfaces, merge points, orchestration logic, model assignments, and eval results. ## Output contract - Full job decomposition: subagent jobs, boundaries, and interfaces - Interface schemas: data format for each handoff - Merge point design: conflict resolution and failure handling per merge - Orchestration layer: routing, sequencing, error handling, approval gates - Model assignments per subagent - Independent eval results per subagent - End-to-end eval results for the full orchestration - Cost comparison: decomposed vs. single-agent ## Failure modes to avoid - **Over-decomposing.** If splitting the job adds more coordination complexity than it saves in execution efficiency, don't split. 3-5 subagents is usually the right range. 15 is almost always wrong. - **Vague interfaces.** "The researcher passes findings to the writer" is not an interface. "The researcher outputs a JSON object with fields: competitors (array of names), market_size (string), key_insights (array of strings)" is an interface. - **Fat orchestrator.** If the orchestrator is doing domain reasoning, it's a single-agent design with extra steps. The orchestrator should only route, merge, and escalate. - **No failure isolation.** If one subagent crashes and the whole workflow dies, you've built a distributed single point of failure. Every subagent failure should have a defined fallback. - **Ignoring cost.** Three model calls at different sizes can be cheaper than one giant call — or they can be 3x more expensive. Measure before assuming decomposition saves money. ## Handoff cues - State the decomposition: subagent count, jobs, and interfaces — another operator needs the architecture overview. - Name the eval results for each subagent and the full orchestration. - Flag any merge points where conflict resolution is still ambiguous and needs human design. - Note the cost comparison so the next operator can decide if decomposition is worth it. ## Example invocation - Slash: `/ai_subagent_decomposition <task>` - Natural language: "Use AI Subagent Decomposition to split this job into focused subagents with clear interfaces." - Example: "This agent is doing research, writing, and review — split it into three specialists." - Example: "The single agent can't handle this workflow within its context window — decompose it." - Often paired with: `ai_agent_spec`, `ai_model_routing`, `ai_eval_harness`, `ai_context_budget` ## Quality bar - A good decomposition makes each subagent's job so clear that another operator could implement it independently. - Every interface must have a defined schema — no "pass results" handwaves. - The merge points must have defined conflict resolution — no "figure it out at runtime." - The total cost and latency of the decomposed system must be measured and compared to the single-agent alternative. ## Related workflows - **Multi-agent build:** `ai_agent_spec` (full job) → `ai_subagent_decomposition` (split) → `ai_agent_spec` (per subagent) → `ai_model_routing` (assign models) → `ai_eval_harness` (test) - **Agent scaling:** single agent → context window pressure → `ai_subagent_decomposition` → `ai_context_budget` (per subagent) → `ai_eval_harness` (end-to-end) - **Cost optimization:** `ai_model_routing` (identify over-provisioned calls) → `ai_subagent_decomposition` (split to right-size) → `ai_eval_harness` (verify quality)
