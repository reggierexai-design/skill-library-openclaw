--- name: ai_prompt_regression_review description: "Review prompt changes for behavior regressions, instruction conflicts, and hidden tradeoffs." user-invocable: true disable-model-invocation: true metadata: {"openclaw":{"emoji":"\ud83e\ude84"}} --- # AI Prompt Regression Review ## Purpose - Review prompt changes for behavior regressions, instruction conflicts, and hidden tradeoffs. - Every prompt change is a risk. A new instruction can conflict with an existing one, a removed constraint can open a safety gap, and a "small" wording change can silently shift the agent's behavior on edge cases. This skill catches those regressions before they reach production. ## Use when - Use before or after changing system prompts, routing rules, or skill instructions. - Use when merging prompt changes from multiple contributors — conflicts hide in the intersection. - Use when the agent's behavior has changed unexpectedly and you need to trace the cause. ## Avoid when - Do not use when prompt behavior has not materially changed — reviewing unchanged prompts is busywork. - If the change is purely stylistic (formatting, whitespace), this is overkill. - If the task is to design a new prompt from scratch, use `ai_prompt_system` instead. ## Inputs to gather - The before and after prompt versions — full text, not summaries. - The intended change: what was supposed to improve, and why. - Eval baseline: quality scores for the current prompt, so you can measure regression. - Known failure modes: past incidents or edge cases that the current prompt handles correctly (and that changes might break). - The agent's full capability surface: tools, memory, routing — changes to one part of the prompt can affect others. ## Operating rules - **Diff, don't rewrite.** Show exactly what changed between the before and after versions. Flag every addition, removal, and modification. A vague "updated the safety section" is not a review. - **Check for instruction conflicts.** When two instructions contradict each other, the model resolves the ambiguity unpredictably. Every new instruction must be checked against every existing instruction. - **Trace the behavioral impact.** For each change, ask: "What does this make the agent more likely to do? Less likely to do? What edge case does this affect that the author didn't intend?" - **Test before trusting.** Run the eval suite against both versions. If you don't have an eval suite, building one is prerequisite — review without measurement is just opinion. - **Watch for priority shifts.** Adding a new instruction can subtly demote existing ones. The model's "attention budget" is finite; adding weight to one instruction means less weight for others. ## OpenClaw tool pattern - Read the full before and after prompts side by side. Highlight every diff. - Run the eval suite against both versions and compare scores. If there's no eval suite, flag this as a critical gap. - Check tool call patterns: does the change affect which tools the agent uses, or how it uses them? - After the review, update the eval baseline if the change is approved. ## Expanded workflow 1. **Diff the prompts.** Generate a precise before/after comparison. List every change: additions, removals, modifications, reordering. 2. **Classify each change.** Tag as: constraint change (what the agent must/mustn't do), behavior change (what the agent does differently), style change (how the agent communicates), or structural change (how the prompt is organized). 3. **Trace behavioral impact.** For each constraint and behavior change, predict: what the agent will now do more of, less of, or differently. Check against known edge cases. 4. **Check for conflicts.** Does any new instruction contradict an existing one? Does any removal unguard a previously protected behavior? Flag conflicts explicitly. 5. **Check priority shifts.** Has the instruction ordering changed? Has the prompt grown (reducing attention per instruction)? Flag any priority shifts that could demote safety-critical rules. 6. **Run the eval comparison.** Execute the eval suite against both versions. Record: which test cases improved, which regressed, which were unchanged. 7. **Produce the review.** Compile: changes, classifications, behavioral predictions, conflicts, eval results, and the overall recommendation (approve / approve with conditions / reject). 8. **If approved, update the baseline.** Record the new eval scores as the baseline for future reviews. ## Output contract - Precise diff between before and after prompts - Change classification: constraint / behavior / style / structural - Behavioral impact predictions per change - Conflict flags: any instructions that now contradict or override each other - Eval comparison: scores before and after, test cases that regressed - Overall recommendation: approve / approve with conditions / reject - Conditions for approval (if any): additional evals needed, safety review required, monitoring needed ## Failure modes to avoid - **Approving without eval.** "Looks fine to me" is not a review. If there's no eval data, the review must flag this as a gap. - **Missing instruction conflicts.** Two instructions that seem compatible in isolation can conflict in practice. Check combinations, not just individual lines. - **Ignoring priority shifts.** A new instruction that pushes the prompt over the model's effective attention length can silently degrade all other instruction following. - **Reviewing the author's intent instead of the actual change.** The change may not do what the author intended. Review what it actually does. - **Cherry-picking test cases.** Only running tests on the happy path proves nothing. The regression is almost always on the edge case. ## Handoff cues - State the overall recommendation and the evidence supporting it. - Name any specific test cases that regressed, and the behavioral prediction for why. - Flag any changes that need `ai_guardrail_review` before deployment. - Provide the updated eval baseline if the change is approved. ## Example invocation - Slash: `/ai_prompt_regression_review <task>` - Natural language: "Use AI Prompt Regression Review to check whether this prompt change breaks anything." - Example: "I changed the safety section — review the diff for regressions and conflicts." - Example: "After this prompt change, the agent stopped calling the right tool — find the regression." - Often paired with: `ai_prompt_system`, `ai_eval_harness`, `ai_guardrail_review`, `ai_failure_taxonomy` ## Quality bar - A good regression review finds at least one issue the change author missed. - Every behavioral prediction must be tested against the eval suite — predictions without evidence are just guesses. - The recommendation must be backed by data: eval scores, conflict analysis, and priority assessment. - If the review finds zero issues and the eval shows no regressions, either the change was trivial or the review was shallow. Verify which. ## Related workflows - **Prompt change cycle:** `ai_prompt_system` (change) → `ai_prompt_regression_review` (review) → `ai_eval_harness` (verify) → deploy or rollback - **Post-incident review:** incident → `ai_failure_taxonomy` (classify) → `ai_prompt_system` (fix) → `ai_prompt_regression_review` (check for new regressions) → `ai_eval_harness` (verify) - **Multi-contributor prompt:** changes from multiple sources → `ai_prompt_regression_review` (merge review + conflict check) → `ai_eval_harness` (integrated test)
