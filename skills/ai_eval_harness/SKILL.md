# AI Eval Harness ## Purpose - Design test cases and score rules that catch regressions in agent quality and safety. - The eval harness is the difference between "I think it works" and "I can prove it works and prove it still works after changes." Without evals, every prompt change is a guess. ## Use when - Use when an agent or prompt needs repeatable evaluation instead of anecdotal impressions. - Use before and after any prompt, tool, or memory change that affects agent behavior. - Use when deploying a new model or changing model routing. - Use when onboarding a new agent — build the eval suite before you build the agent. ## Avoid when - Do not declare quality from a handful of cherry-picked transcripts — that's marketing, not eval. - If the task is to design the agent itself, start with `ai_agent_spec` first; eval comes after the spec exists. - Do not use this as a substitute for real user feedback — eval catches regressions, not misalignment with what users actually want. ## Inputs to gather - The agent's job: what decision or action it owns, and what "done right" looks like. - Current prompt, tools, and runtime config — the eval must test the actual system, not a description of it. - Known failure modes: past incidents, user complaints, hallucination patterns, tool misuse. - Quality dimensions that matter: correctness, safety, helpfulness, conciseness, tool use accuracy. - Available eval infrastructure: can you run automated tests, or is manual review required? - Pass threshold: what fraction of test cases must pass for the agent to be considered production-ready? ## Operating rules - **Evaluate behaviors that matter, not just style.** A beautifully formatted wrong answer is still wrong. Score correctness and safety before style. - **Cover the full triad: happy path, edge cases, adversarial inputs.** An eval suite with only easy cases will never catch the failures that embarrass you in production. - **Define score rules before reading results.** If you score after seeing the output, you'll rationalize failures. Write the rubric, run the eval, accept the score. - **Make evals cheap to run.** If running the full suite takes hours and dollars, it won't get run. Prioritize fast, cheap, automated over comprehensive but prohibitive. - **Version the eval suite alongside the prompt.** When the prompt changes, the eval suite may need to change too. Track them together. ## OpenClaw tool pattern - Read the current prompt, tool schema, and runtime config before designing evals. An eval that doesn't match the real system is worthless. - Run evals against the actual agent endpoint, not against the model alone. Tool use, memory, and routing all affect behavior. - Store eval results alongside the prompt version in the workspace. When a regression appears, you need to know which change caused it. - Use `ai_prompt_regression_review` to compare eval results across prompt versions. ## Expanded workflow 1. **Define target behaviors and failure modes.** List what the agent should do, what it should never do, and what it does ambiguously. Use `ai_failure_taxonomy` if failure modes are unclear. 2. **Write test cases for each behavior.** For each target behavior, write: a representative input, the expected behavior, and a scoring rubric (pass/fail or 1-5 scale). Include at least: 3 happy path cases, 3 edge cases, 3 adversarial inputs. 3. **Set scoring rules and pass thresholds.** Define how each test case is scored. Set the overall pass threshold (e.g., "90% of test cases pass, 0 safety failures"). Define what constitutes a regression (e.g., "score drops more than 5% from baseline"). 4. **Build the harness.** Create a script or config that runs all test cases against the agent, collects outputs, and scores them. Make it runnable with a single command. 5. **Run the baseline.** Execute the full eval suite against the current agent. Record scores as the baseline. Any future change is judged against this baseline. 6. **Design the maintenance loop.** Define when the eval suite runs (every prompt change, every deployment, every week). Define who reviews failures and how they get fixed. 7. **Document the eval strategy.** Write a short eval plan: what's tested, how it's scored, the baseline, the maintenance loop, and known gaps. 8. **End with the eval command.** The output should include the exact command or workflow to re-run the eval suite. ## Output contract - Target behaviors and failure modes tested - Test case set with inputs, expected outputs, and scoring rubrics - Baseline scores from initial run - Pass thresholds and regression criteria - Eval execution command or workflow - Maintenance schedule and review process - Known gaps: behaviors not yet tested, edge cases not yet covered ## Failure modes to avoid - **Evaluating style over substance.** A rubric that scores "tone" but not "correctness" will approve confident wrong answers. - **Cherry-picking test cases.** An eval suite of only easy cases proves nothing. Include the cases you're afraid of. - **Scoring after seeing results.** If you adjust the rubric to make a failing test pass, you've defeated the purpose. Define the rubric first, accept the score. - **Never re-running the eval.** An eval suite that's run once and forgotten is a sunk cost. Build the maintenance loop or don't bother building the eval. - **Testing the prompt in isolation.** An agent is a system. If you only test the prompt without tools, memory, and routing, you're testing a simplified version that won't match production behavior. ## Handoff cues - State the baseline scores and the pass threshold clearly. - Name the eval command so the next operator can re-run independently. - Flag any test cases that require manual review (adversarial inputs, subjective quality judgments). - Note any behaviors that couldn't be automated and need real user feedback. ## Example invocation - Slash: `/ai_eval_harness <task>` - Natural language: "Use AI Eval Harness to build a test suite that catches regressions before users do." - Example: "Design an eval suite for this agent — I need to know when changes break things." - Example: "Score this agent's outputs against a rubric so I can compare prompt versions." - Often paired with: `ai_prompt_system`, `ai_prompt_regression_review`, `ai_guardrail_review`, `ai_failure_taxonomy` ## Quality bar - A good eval harness catches real regressions — not hypothetical ones, and not just style nitpicks. - The output must include a runnable eval command, not just a description of what should be tested. - If the eval suite can't be run in under 10 minutes for under $1, it's too heavy. - At least 10 test cases covering happy path, edge cases, and adversarial inputs — fewer means the suite is likely incomplete. ## Related workflows - **Prompt change cycle:** `ai_prompt_system` (change) → `ai_eval_harness` (verify) → `ai_prompt_regression_review` (compare) - **Agent deployment:** `ai_agent_spec` → `ai_prompt_system` → `ai_eval_harness` (baseline) → `ai_guardrail_review` → deploy - **Failure investigation:** `ai_failure_taxonomy` (classify) → `ai_eval_harness` (add test case) → `ai_prompt_system` (fix) → `ai_eval_harness` (re-verify) - **Safety audit:** `ai_guardrail_review` → `ai_eval_harness` (adversarial suite) → `sec_threat_model` (system-level review)
