# AI Failure Taxonomy ## Purpose - Classify agent failures so fixes target the real weakness instead of symptoms. - When an agent fails, the visible behavior (wrong answer, hallucinated fact, missed tool call) is a symptom. This skill traces the symptom to its root cause category, so the fix addresses the actual weakness instead of papering over it with prompt tweaks. ## Use when - Use after bad runs, hallucinations, weak outputs, or missed tool use. - Use when the same failure pattern keeps recurring despite prompt changes — this usually means you're fixing the wrong layer. - Use when onboarding a new agent and you need to anticipate likely failure modes before they happen. ## Avoid when - Do not use when there is no concrete failure evidence to classify — this isn't theoretical, it's diagnostic. - If the failure is clearly a bug in the tool or API (wrong return value, timeout), route to `eng_bug_triage` instead. - If the failure is a safety violation (PII leak, harmful output), route to `ai_guardrail_review` immediately — classify after containing. ## Inputs to gather - The concrete failure: what the agent did vs. what it should have done. - Full transcript of the failing run, not just the bad output — context often reveals where the reasoning went wrong. - Current prompt, tools, and memory config — the classification depends on the system design. - Frequency and pattern: is this a one-off or a recurring failure? Does it happen on specific inputs, at specific conversation lengths, or with specific tools? ## Operating rules - **Classify by root cause layer, not by surface symptom.** The six layers: (1) prompt — missing or ambiguous instructions, (2) context — wrong, stale, or insufficient information, (3) tool — wrong tool selected, wrong parameters, misinterpreted output, (4) model — capability gap, reasoning failure, or training bias, (5) memory — stale state, forgotten constraints, cross-session contamination, (6) architecture — wrong routing, missing fallback, unconstrained autonomy. - **Trace the causal chain, not just the endpoint.** "The agent gave a wrong answer" is the endpoint. The chain might be: context retrieval returned stale docs → agent trusted retrieved content over instructions → no verification step → wrong answer. The fix goes on the first broken link. - **Distinguish systematic from stochastic failures.** A failure that reproduces on the same input is systematic (fixable). A failure that only happens sometimes is stochastic (requires guardrails, not fixes). - **Match the fix to the layer.** Prompt problems → rewrite instructions. Context problems → fix retrieval or gating. Tool problems → fix schema, add validation. Model problems → route to a stronger model or add verification. Memory problems → fix the memory policy. Architecture problems → redesign the decision loop. - **Never fix a layer 3+ problem with a layer 1 solution.** Adding "always double-check tool outputs" to the prompt is a band-aid. Adding a verification step in the architecture is a fix. ## OpenClaw tool pattern - Read the full failing transcript, not just the bad turn. The context window state at the time of failure matters. - Check the tool call logs to see what the agent requested vs. what it received. - Review the memory state if the failure involves cross-session behavior. - After classification, propose the fix at the correct layer and add a test case to `ai_eval_harness` to prevent regression. ## Expanded workflow 1. **Capture the failure.** Document the concrete failure: input, expected output, actual output, and the full context leading up to the failure. 2. **Map the causal chain.** Trace backward from the bad output: what did the agent reason? What context did it use? What tools did it call? Where did the reasoning first go wrong? 3. **Classify by root cause layer.** Assign the primary failure to one of the six layers: prompt, context, tool, model, memory, architecture. If multiple layers contributed, name each and rank by impact. 4. **Distinguish systematic from stochastic.** Can you reproduce the failure on the same input? If yes, it's systematic. If no, it's stochastic and needs guardrails, not a fix. 5. **Propose the fix at the correct layer.** Match the fix to the root cause. Name the specific change: which instruction to rewrite, which retrieval to fix, which tool schema to update, which model to route to, which memory policy to change, or which architectural step to add. 6. **Add a regression test.** Create a test case for `ai_eval_harness` that would catch this failure if it recurs. The test case is as important as the fix. 7. **Document the pattern.** If this is a recurring pattern, add it to a failure pattern library so future incidents can be classified faster. ## Output contract - Failure description: concrete input, expected output, actual output - Causal chain: step-by-step trace of where reasoning went wrong - Root cause classification: primary layer + contributing layers - Fix proposal: specific change at the correct layer - Regression test case for `ai_eval_harness` - Failure pattern (if recurring): name, description, root cause layer, prevention strategy ## Failure modes to avoid - **Classifying by surface symptom.** "The agent hallucinated" is not a root cause. WHY did it hallucinate? Missing context? Bad retrieval? No verification step? The classification must go deeper. - **Fixing the wrong layer.** Adding a prompt instruction to "be more careful" is not a fix for a tool schema problem or an architecture gap. Match the fix to the layer. - **Treating stochastic failures as systematic.** If the failure only happens sometimes, a prompt fix won't prevent it. Add guardrails or verification steps instead. - **Skipping the regression test.** A fix without a test case is a fix that will silently break. Always add a test case. - **Over-classifying.** Not every failure needs a six-layer analysis. A simple prompt bug gets a simple fix. Reserve deep analysis for recurring or high-impact failures. ## Handoff cues - State the root cause layer and the proposed fix clearly. - Name the test case that should be added to the eval suite. - If the fix requires changes to tools, memory, or architecture, flag which other skills should be involved (`ai_agent_spec`, `ai_tool_schema_review`, `ai_memory_policy`). - If the failure has safety implications, flag it for `ai_guardrail_review`. ## Example invocation - Slash: `/ai_failure_taxonomy <task>` - Natural language: "Use AI Failure Taxonomy to classify why the agent failed and where the fix should go." - Example: "The agent keeps giving wrong answers — classify the failure so I can fix the right thing." - Example: "This hallucination keeps recurring despite prompt changes — what layer is the real problem?" - Often paired with: `ai_eval_harness`, `ai_prompt_system`, `ai_guardrail_review`, `ai_agent_spec` ## Quality bar - A good failure classification names the root cause layer, not just the surface symptom. - The proposed fix must target the correct layer — if it's just a prompt band-aid for an architecture problem, the classification isn't done. - The output must include a regression test case. No exceptions. ## Related workflows - **Failure investigation:** `ai_failure_taxonomy` (classify) → layer-specific fix (`ai_prompt_system` / `ai_tool_schema_review` / `ai_memory_policy` / `ai_agent_spec`) → `ai_eval_harness` (add test + verify) - **Safety incident:** `ai_guardrail_review` (contain) → `ai_failure_taxonomy` (classify) → fix → `ai_eval_harness` (regression test) - **Recurring pattern:** `ai_failure_taxonomy` (multiple incidents) → `ai_agent_spec` (redesign to structurally prevent) → `ai_eval_harness` (comprehensive suite)
