# Changelog Story ## Purpose - Turn product updates into clear stories that reinforce momentum, quality, and user trust. - Most changelogs are boring lists of technical changes. This skill reframes each update as a user-facing story: what changed, why it matters, and what the user should do next. The goal is cumulative trust — each update should make users feel the product is actively improving for them. ## Use when - Use for release notes, product update posts, launch emails, or roadmap communication. - Use when the team shipped something and needs to communicate it beyond a bullet list. - Use when users are asking "is this product still being worked on?" ## Avoid when - Do not use when there is no meaningful user-facing change to communicate — "internal refactoring" with no user impact doesn't need a story. - If the update is a single minor fix, a short note beats a dressed-up story. - Do not use to spin bad news (outages, data loss) — use `ops_status_update` for that. ## Inputs to gather - What shipped: the actual changes, in plain language. - Who it affects: which user segments, which use cases. - Why it matters: what problem it solves or what it enables. - The proof: performance data, before/after examples, or user feedback that validates the change. - Channel: changelog page, email, in-app notification, social post, or all of the above. ## Operating rules - **Lead with what changed for the user, not what changed in the code.** "You can now export to CSV" beats "Added CSV export endpoint to /api/v2/exports." - **Explain why it matters, not just what shipped.** Every feature answers a question or solves a pain. Name it. "No more copy-pasting data into spreadsheets" beats "CSV export now available." - **Keep the tone grounded and specific.** Momentum comes from consistency, not hype. A steady drumbeat of real improvements builds more trust than occasional overblown announcements. - **Group by user impact, not by engineering team.** Users don't care which team built what. They care about their workflow. Group changes by theme: "Faster workflows," "New integrations," "Bug fixes." - **Include migration notes when needed.** If something changed that requires user action, say it clearly, early, and with specific steps. Hiding breaking changes is how you lose trust. ## OpenClaw tool pattern - Read the actual PR descriptions, commit messages, or release notes — don't rely on memory of what shipped. - Ground the narrative in real user feedback or support tickets that motivated the change. - When the update touches existing docs, flag what needs updating in `doc_release_notes` or `doc_migration_guide`. ## Expanded workflow 1. **Inventory the changes.** List every user-facing change. Group by impact level: major (new capability), moderate (significant improvement), minor (quality-of-life), fix (bug resolved). 2. **Translate each change to user language.** For each: what can the user do now that they couldn't before? What's faster, easier, or better? What pain is gone? 3. **Structure the story.** Framework: Headline (the biggest win for users) → What Changed (grouped by theme, user-facing language) → Why It Matters (the pain it solves or the door it opens) → What To Do Next (any migration steps, opt-in settings, or CTA). 4. **Write the changelog.** Match the tone to the audience: developer changelogs can be technical; user-facing updates should be plain language. 5. **Create channel derivatives.** Full changelog for the site. Short summary for email. Thread for social. In-app one-liner for notification. 6. **Review for honesty.** Did you overclaim? Did you bury a breaking change? Would a power user feel misled? ## Output contract - Headline: the single biggest user-facing win - What Changed: grouped by theme, in user language - Why It Matters: the pain solved or opportunity enabled - Migration notes: any user action required - Channel derivatives: full changelog, email summary, social thread, in-app notification ## Failure modes to avoid - **Burying breaking changes.** If something requires user action, it goes at the top, not buried in a "minor changes" section. - **Hype on minor changes.** Calling a bug fix "revolutionary" trains users to ignore your announcements. Match the language to the impact. - **Jargon in user-facing updates.** "Refactored the data pipeline" means nothing to users. "Reports load 3x faster" means everything. - **Inconsistent cadence.** Publishing 5 updates in a week then going silent for 2 months erodes the momentum signal. Set a sustainable rhythm. ## Handoff cues - State where the changelog will be published and which derivatives are ready. - Flag any breaking changes that need additional communication (email blast, in-app banner). - Note any docs that need updating to reflect the changes. ## Example invocation - Slash: `/att_changelog_story <task>` - Natural language: "Use Changelog Story to turn these release notes into something users will actually read." - Example: "We shipped 3 features and 5 fixes this sprint — write the changelog." - Often paired with: `doc_release_notes`, `att_content_calendar`, `ops_status_update` ## Quality bar - A good changelog story makes shipping feel real and cumulative. A user should finish it thinking "this product keeps getting better for me." - Every change must be in user language, not engineering language. - Breaking changes must be prominent, not buried. - The tone should match the impact: big wins get enthusiasm, bug fixes get matter-of-fact clarity. ## Related workflows - **Release communication:** `doc_release_notes` (internal) → `att_changelog_story` (user-facing) → `att_content_calendar` (schedule) → `att_email_nurture_sequences` (notify users) - **Launch day:** `att_launch_plan` → `att_changelog_story` (announce) → `att_thread_writer` (social) → `ops_status_update` (internal) - **Ongoing trust:** regular shipping → `att_changelog_story` (each release) → cumulative momentum signal
